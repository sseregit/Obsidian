```text
패키지 전체를 분석해 테스트 작성에 불리한 구조(복잡한 의존, 중복/고아 코드, 부수효과 등)를 식별한다.
    - 문제점을 기반으로 Todo 리스트를 작성하고, 필요한 부분만 선별적으로 리팩터링한다.
    - 리팩터링이 불필요한 경우 “스킵 사유”를 명확히 설명한다.
    - 리팩터링은 기능 변경 없이 진행하며, 메서드/클래스 추출 및 구조 정리에 집중해 테스트 친화성을 높인다.
    - 메서드명·변수명·클래스명은 Clean Code 원칙에 맞게 개선한다.
    - 필요한 경우에만 짧고 명확한 한글 주석을 추가한다.
    - 테스트 코드는 작성하지 않으며, 별도 요청 시에만 제공한다.
    - 커밋 전에 불필요한 의존성, 고아 코드, 사용되지 않는 Bean/클래스 제거 여부를 확인한다.
    - 가능하면 빌드까지 수행해 구조적 문제를 검증하되, 테스트 실행은 생략한다. 제한이 있을 경우 사유를 명시한다.
    - 커밋 메시지는 한글로 제안한다.
    - SonarQube 규칙을 참고해 개선 포인트(Code Smell, Bug, Security Hotspot)를 함께 반영한다.
    - Checkstyle/PMD/SpotBugs/Google Java Format 등 Java 정적 분석·포맷팅 권장사항을 준수한다.
```
- 목표
	- 현재 구조를 최대한 건드리지 않으면서 추출하는 전략을 사용해 추출해서 각 메서드, 클래스의 볼륨을 줄인다.
	- 현재 프로젝트는 테스트 코드가 없고 테스트 코드를 작성하기 전에 전체적으로 프로젝트를 정리 하기 위한 프롬프트 작성
- 키워드
	- Clean Code, SonarQube, Code Smell, Bug, Security Hotspot....
- 결과
	- 리팩토링의 기능 변경없이 진행과 추출 및 구조 정리에 집중한다는 프롬프트가 잘 적용이된거 같다. **많은 수정이 이루어져도 기능에 문제가 없는 점이 좋다.**
